# 透明度重构方案

## 一、透明度定义

本项目中的“透明度”指：

1. 接口透明
- 谁在调用、调用了什么版本、成功率如何、耗时如何

2. 数据透明
- 指标如何计算、口径何时变更、旧新差异是否可追踪

3. 运行透明
- 系统在何处慢、何处错、为何错、影响范围多大

4. 变更透明
- 每次上线改了什么、风险是什么、回滚路径是什么

## 二、现状与差距

现状具备：
- Prometheus 指标基础
- Correlation ID (`X-ID`)

缺口：
- 缺统一日志规范（结构化字段）
- 缺 trace 全链路
- 缺 API 版本迁移看板
- 缺指标字典和口径变更审计

## 三、实施方案

### 1. 观测标准化

1. 统一日志字段
- `trace_id`, `request_id`, `user_id`, `company_id`, `vessel_id`, `endpoint`, `latency_ms`, `result_code`

2. Trace 接入
- 前端 `x-request-id` 透传
- 后端接 OpenTelemetry（HTTP/DB/Redis/外部API）

3. 业务事件埋点
- 上传成功/失败
- 优化计算启动/完成/失败
- 路径规划命中缓存/未命中

### 2. API 契约透明化

1. OpenAPI 版本化
- 固定生成 `openapi-v1.json`, `openapi-v2.json`

2. 变更审计
- 每次发布自动跑 openapi diff
- 对 breaking change 触发阻断

3. 兼容观察
- 旧接口调用占比
- 迁移覆盖率

### 3. 数据口径透明化

1. 建立指标字典
- CII、油耗、航速、吃水差、状态评分等

2. 口径变更流程
- 变更申请 -> 评审 -> 对账 -> 发布

3. 结果可解释
- 算法输出附关键输入、参数与版本号

## 四、透明度交付物

1. 运行看板
- API可用性、P95延迟、错误率、缓存命中率

2. 业务看板
- 上传处理时长、优化任务耗时分布、失败原因TOP

3. 变更看板
- 发布变更列表、契约差异、回滚记录
