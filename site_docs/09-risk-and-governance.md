# 风险与治理

## 主要风险

1. 功能回归风险
- 原因：模块多、接口多、历史兼容复杂
- 对策：功能矩阵 + 自动化回归 + 灰度发布

2. 性能收益不达预期
- 原因：未抓住真正热点
- 对策：先基线，再优化，按数据决策

3. 数据口径偏移
- 原因：统计/模型逻辑调整
- 对策：指标字典 + 对账报告 + 容差治理

4. 团队协作风险
- 原因：多模块并行改造冲突
- 对策：分层 ownership + RFC 机制 + 每周评审

## 治理机制

1. 变更治理
- 架构变更需 RFC
- Breaking change 需双人评审

2. 发布治理
- 发布前完成契约 diff 与压测
- 发布后 24h 重点监控

3. 文档治理
- 文档与代码同 PR
- 每个里程碑输出阶段总结

## 角色建议

1. 产品负责人：功能保留与验收标准
2. 架构负责人：重构路径与技术决策
3. 平台负责人：CI/CD、可观测与发布治理
4. 模块负责人：具体模块改造与质量
