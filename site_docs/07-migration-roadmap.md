# 迁移路线图

## 总体策略

采用 4 阶段渐进式迁移，确保持续交付且不影响线上功能。

## Phase 0：基线建立（1-2 周）

1. 冻结现有功能范围
2. 建立功能保留矩阵
3. 建立性能基线与透明度基线
4. 搭建文档、看板、CI门禁

产出：
- 功能清单 v1
- 基线报告 v1
- 契约快照 v1

## Phase 1：兼容层与观测先行（2-3 周）

1. API 兼容路由与 deprecation 机制
2. 统一请求 ID 透传
3. OpenTelemetry + 结构化日志上线
4. 契约 diff 自动化

产出：
- 兼容层上线
- 观测闭环建立

## Phase 2：性能核心改造（3-5 周）

1. 前端请求层重构 + 图表渲染优化
2. 后端查询优化 + 缓存策略升级
3. 航线优化任务异步化
4. 压测门禁纳入 CI

产出：
- 性能优化版本
- 压测报告

## Phase 3：收口与下线（2-3 周）

1. 旧路径调用量降到目标阈值
2. 下线冗余兼容逻辑
3. 完成最终回归与验收

产出：
- 稳定版 release
- 迁移总结报告

## 里程碑门禁

1. M1：功能零回归通过
2. M2：关键接口 P95 达标
3. M3：观测覆盖达标
4. M4：旧接口调用降至目标
