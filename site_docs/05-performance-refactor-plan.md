# 性能重构方案

## 一、性能问题分层

### 前端

1. 请求层
- Promise 手工封装导致错误路径冗余，重试和并发控制困难

2. 渲染层
- 图表组件渲染开销大
- 大量页面在切换/监听时重复请求

3. 资源层
- 缺少按路由拆分与资源优先级策略

### 后端

1. IO层
- 部分接口依赖数据库与外部服务，失败重试、连接池策略仍可优化

2. 计算层
- 航线优化相关图构建与气象计算非常重

3. 缓存层
- 目前有缓存，但缺“命中率、TTL、失效策略、热点保护”统一治理

## 二、优化目标（建议）

1. 前端
- 首屏 LCP < 2.5s（内网条件可按实际调整）
- 关键页面交互响应 < 200ms
- 图表切换卡顿显著下降

2. 后端
- 关键读接口 P95 < 300ms
- 复杂计算接口提供异步任务模式，在线请求 P95 可控

## 三、实施方案

### 前端改造清单

1. 统一请求客户端
- 用 typed API client 替换分散式调用
- 标准化超时、重试、取消、错误码映射

2. 数据与渲染解耦
- 图表数据预处理放到 worker 或计算层
- 图表组件支持按需刷新和虚拟化

3. 页面级缓存策略
- 将只读高频接口纳入 SWR/缓存层
- 明确失效条件（船舶切换、时间范围变化）

4. 打包与加载优化
- 路由懒加载
- 大包拆分（echarts 按需）
- 静态资源压缩与缓存头策略

### 后端改造清单

1. 数据访问优化
- repository 查询审计（N+1、无索引、全表扫描）
- 增加关键索引，必要时拆分查询路径

2. 计算任务异步化
- 将重型航线优化任务切入任务队列（如 Celery/RQ）
- API 返回任务 ID，前端轮询/订阅任务状态

3. 缓存体系升级
- Redis 统一 key 规范、TTL 策略、热点保护
- 关键接口记录命中率并进入看板

4. 容量与压测
- 按模块建立基线压测（upload/statistic/optimization/route-optimization）
- 将压测纳入发布门禁

## 四、优先级

1. P0
- 请求层统一
- DB 查询热点优化
- 航线优化异步化设计

2. P1
- 图表渲染与资源加载优化
- 缓存命中率治理

3. P2
- 深度算法与模型推理加速
- 长期容量规划
