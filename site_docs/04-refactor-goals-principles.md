# 重构目标与原则

## 目标

1. 性能
- 前端首屏与关键页面交互延迟显著下降
- 后端核心接口 P95 延迟下降
- 重型计算任务对在线请求影响可控

2. 透明度
- 接口、指标、日志、告警、版本演进可追踪
- 重构过程每一步都有可量化验收证据

3. 功能零回归
- 现有功能全保留
- 对外契约兼容，迁移过程可平滑切换

## 重构原则

1. 兼容优先
- 先做兼容层（BFF/Adapter/版本路由），再做内部替换

2. 渐进替换
- Strangler Fig 模式：先旁路新实现，再灰度切流，最后下线旧实现

3. 数据可核对
- 每个改造点都能给出旧新对账报告

4. 可观测先行
- 没有监控和可视化，不上线关键重构

5. 性能基线先建立
- 没有基线与压测，不优化关键路径

## 架构重构蓝图

### 前端目标架构

1. `apps/web`（UI）
2. `packages/api-client`（类型化请求层）
3. `packages/state`（状态管理）
4. `packages/charts`（图表能力）
5. `packages/observability`（前端埋点/日志）

### 后端目标架构

1. API 层：FastAPI 路由 + OpenAPI 契约版本控制
2. 应用层：明确 use-case 服务（去粘连）
3. 领域层：指标计算与优化算法边界清晰
4. 基础设施层：DB/缓存/任务队列/对象存储
5. 可观测层：Logs/Metrics/Traces/Events

### 兼容层策略

1. 保留旧路径，通过 adapter 转发
2. 统一返回模型，逐步引入 v2 模式
3. 提供迁移仪表盘（旧接口调用量、错误率、替换进度）
