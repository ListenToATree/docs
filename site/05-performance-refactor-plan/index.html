    性能重构方案¶

    一、性能问题分层¶

    前端¶

    1. 请求层

    2. Promise 手工封装导致错误路径冗余，重试和并发控制困难

    3. 渲染层

    4. 图表组件渲染开销大

    5. 大量页面在切换/监听时重复请求

    6. 资源层

    7. 缺少按路由拆分与资源优先级策略

    后端¶

    1. IO层

    2. 部分接口依赖数据库与外部服务，失败重试、连接池策略仍可优化

    3. 计算层

    4. 航线优化相关图构建与气象计算非常重

    5. 缓存层

    6. 目前有缓存，但缺“命中率、TTL、失效策略、热点保护”统一治理

    二、优化目标（建议）¶

    1. 前端
    2. 首屏 LCP < 2.5s（内网条件可按实际调整）
    3. 关键页面交互响应 < 200ms

    4. 图表切换卡顿显著下降

    5. 后端

    6. 关键读接口 P95 < 300ms
    7. 复杂计算接口提供异步任务模式，在线请求 P95 可控

    三、实施方案¶

    前端改造清单¶

    1. 统一请求客户端
    2. 用 typed API client 替换分散式调用

    3. 标准化超时、重试、取消、错误码映射

    4. 数据与渲染解耦

    5. 图表数据预处理放到 worker 或计算层

    6. 图表组件支持按需刷新和虚拟化

    7. 页面级缓存策略

    8. 将只读高频接口纳入 SWR/缓存层

    9. 明确失效条件（船舶切换、时间范围变化）

    10. 打包与加载优化

    11. 路由懒加载
    12. 大包拆分（echarts 按需）
    13. 静态资源压缩与缓存头策略

    后端改造清单¶

    1. 数据访问优化
    2. repository 查询审计（N+1、无索引、全表扫描）

    3. 增加关键索引，必要时拆分查询路径

    4. 计算任务异步化

    5. 将重型航线优化任务切入任务队列（如 Celery/RQ）

    6. API 返回任务 ID，前端轮询/订阅任务状态

    7. 缓存体系升级

    8. Redis 统一 key 规范、TTL 策略、热点保护

    9. 关键接口记录命中率并进入看板

    10. 容量与压测

    11. 按模块建立基线压测（upload/statistic/optimization/route-optimization）
    12. 将压测纳入发布门禁

    四、优先级¶

    1. P0
    2. 请求层统一
    3. DB 查询热点优化

    4. 航线优化异步化设计

    5. P1

    6. 图表渲染与资源加载优化

    7. 缓存命中率治理

    8. P2

    9. 深度算法与模型推理加速
    10. 长期容量规划
